# 課題2｜命名とスキーマの正常化

**区分**: 基礎編 ／ **AI利用**: 併用可 ／ **想定時間**: 1日

## 目的
「スキーマとデータの正はマイグレーション/シーダーであり、dump.sqlではない」を体得する。命名の重要性を理解する。

## 🔑 この課題で学ぶLaravel用語
（調べる時のキーワード。「使い方を身につける」対象です）
- **マイグレーション（Migration）** — DBの表構造をコードで定義・変更管理する仕組み
- **モデル（Model / Eloquent）** — 1テーブルに対応するクラス
- **ファクトリ（Factory）** — テスト用ダミーデータの生成器
- **シーダー（Seeder）** — 初期データの投入
- **Artisanコマンド（`php artisan migrate:fresh --seed`）** — 開発用CLI

## 現状の問題
- モデル `Table1`、テーブル `table1` / `table2` という意味不明な命名。
- マイグレーションのファイル名は `create_posts_table` なのに、中身は `table1` を作っている。
- **`table2` を作るマイグレーションが存在しない**（`database/dump.sql` にしか定義がなく、まっさらなDBで `migrate` すると壊れる）。

## やること
1. `Table1` → `Post`、`table1` → `posts` へリネーム（モデル・マイグレーション・参照箇所すべて）。
2. `table2` の役割を definitionから読み取り、意味のある名前（例：`post_details`）で **正式なマイグレーションを作成**。
3. `PostFactory` と `Seeder` を整備し、テストデータを再現可能にする。
4. `php artisan migrate:fresh --seed` だけで動作する状態にする。

## 提出物
- リネーム後のモデル / マイグレーション
- `table2` 相当を作る新規マイグレーション
- Factory / Seeder
- `migrate:fresh --seed` 実行ログ

## 受け入れ基準
- [ ] まっさらなDBで `migrate:fresh --seed` が成功する
- [ ] dump.sql に依存せずアプリが動く
- [ ] テーブル/カラム/モデル名が役割を表している
