# 課題1｜リポジトリ衛生とシークレット管理

**区分**: 基礎編 ／ **AI利用**: 併用可 ／ **想定時間**: 半日

## 目的
「秘密情報・本番データはGitに入れない」という現場の必須ルールを体得する。

## 🔑 この課題で学ぶLaravel用語
（調べる時のキーワード。「使い方を身につける」対象です）
- **`.env` / 環境変数** — 秘密情報・環境依存の設定をコードから分離する仕組み
- **設定ファイル `config/*.php` と `config()` / `env()`** — 設定値の読み出し方
- **（周辺）`.gitignore`** — Gitの追跡から外す指定

## 現状の問題
- `.env` がGit管理下にコミットされている（`.gitignore` の `# .env` がコメントアウトされ無効化されている）。
- `database/dump.sql` に **実在するAPIトークンの本物の値・パスワードハッシュ・メールアドレス** が含まれた状態でコミットされている。

## やること
1. `.gitignore` を修正し、`.env` と `database/dump.sql`（実データを含むダンプ）を管理対象から外す。
2. すでにコミット済みの `.env` / `dump.sql` を `git rm --cached` で追跡解除する。
3. 秘密情報のサンプルは `.env.example` に集約する（実値は入れない）。

> 💡 **補足：dump.sql は「初期構築の代わり」になっている**
> 現状 `dump.sql` は、DBの表や初期データを用意する手段として使われていると思われます。ただしLaravelでは、これは **マイグレーション（表の定義）＋ シーダー／ファクトリ（初期データ投入）** で実現するのが正しいやり方です（`php artisan migrate --seed`）。
> だから `dump.sql` をGitから外しても、**同じ初期構築を「コードで再現可能な正しい形」で置き換えられます。** その具体的な作り方は **[課題2](課題2_命名とスキーマ正常化.md) で学びます**（ここでは「消して大丈夫、後で正攻法に置き換える」と理解できればOK）。

## 提出物
- 修正後の `.gitignore`
- `git ls-files | grep -E "env|dump"` の結果（追跡されていないこと）

## 受け入れ基準

- [ ] `.env.example` に必要なキーが揃っている（値は空/ダミー）
- [ ] なぜ危険だったかを自分の言葉で説明できる
