# 課題8｜イベント・通知・管理者機能

**区分**: 応用編 ／ 🔸**オプション**（13日の必須ではない。予備日が余れば取り組む） ／ **AI利用**: 併用可 ／ **想定時間**: 1.5日

## 目的
副作用（通知・メール）をコントローラーに溜めず**イベント駆動**で疎結合にする。**ミドルウェア**で管理者エリアを保護する。

> 🔸 **この課題はオプションです。** 通知・キューの最小限は卒業課題（課題9）の「予約確認メール」でも体験します。ここは Observer・管理者ミドルウェアまで含めた**深掘り版**なので、時間に余裕があれば取り組んでください。

## 🔑 この課題で学ぶLaravel用語
（調べる時のキーワード。「使い方を身につける」対象です）
- **イベント / リスナー（Event, Listener）** — 発火と購読で処理を疎結合にする
- **オブザーバ（Observer）** — モデルの created / updated 等にフックする
- **通知 / メール（Notification, Mail）** — 「記事公開しました」等の送信
- **キュー連携（`ShouldQueue`）** — 通知・メールを非同期化（画面の応答を遅くしない）
- **ミドルウェア（Middleware）** — リクエスト前後の共通処理（Go/Nextの middleware と同じ発想）。管理者チェックに使う

## やること
1. 記事が公開されたら `PostPublished` **イベント**を発火し、**リスナー**で処理する。
2. 作者へ **通知（メール等）** を送る。**`ShouldQueue` で非同期化**し、`php artisan queue:work` で処理。
3. **管理者だけがアクセスできる管理ページ**を作り、**自作ミドルウェア**（`EnsureUserIsAdmin`）で保護する。
4. テストで **`Notification::fake()` / `Event::fake()`** を使い、送信されたことを検証。

## 提出物
- Event / Listener / Notification（または Mail）クラス
- 管理者チェックのミドルウェアと適用箇所
- 通知が飛ぶことを確認するテスト

## 受け入れ基準
- [ ] 記事公開の副作用がコントローラーから切り離されている
- [ ] 通知・メールが非同期で送られる
- [ ] 管理者以外は管理ページにアクセスできない
