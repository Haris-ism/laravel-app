# 課題5｜認証（Breeze）と認可（Policy）

**区分**: 基礎編 ／ **AI利用**: 🚫 **禁止・自力** ／ **想定時間**: 1.5日

## 目的
「組み込み（画面あり）」での**ログインの仕組み（セッション認証）**を理解し、「誰が何をしてよいか」を**Policy**で表現する。

> 現状のコードは Sanctum の**トークン認証**（API向け）でした。組み込みの画面では **セッション（Cookie）認証**が標準です。ここを切り替えます。

## 🔑 この課題で学ぶLaravel用語
（調べる時のキーワード。「使い方を身につける」対象です）
- **セッション認証 / Laravel Breeze** — ログイン画面つきの標準的な認証（画面ありアプリ向け）
- **`auth` ミドルウェア** — ログイン必須ページの保護
- **フォームリクエスト（FormRequest）** — 入力検証を専用クラスに分離（既存コードにあり）
- **ポリシー（Policy）／ `$this->authorize()`** — 「本人だけ編集・削除できる」等の認可ルール
- **（違いの整理）認証＝本人確認 / 認可＝権限確認**

## 現状の問題
- 認証がトークン方式（API向け）で、画面つきアプリに合っていない。
- 記事に `user_id` がなく、**ログインさえすれば誰でも全記事を編集できる**。
- FormRequest の `authorize()` が `return true` 固定。

## やること
1. **Laravel Breeze**（Bladeスタック）を導入し、**ログイン／登録画面つきのセッション認証**にする。
2. 記事に `user_id` を追加し、投稿者を紐付ける。
3. 記事の作成・編集・削除ページを **`auth` ミドルウェア**で保護する。
4. `PostPolicy` を作り、「**投稿者本人のみ** 編集・削除可」を実装（コントローラーで `authorize()`）。
5. 他人の記事の編集ページに入ろうとすると **403** になることを確認。

## 提出物
- Breeze導入後のログイン/登録画面
- `user_id` を足すマイグレーション
- `PostPolicy` と、認可を組み込んだコントローラー
- 「認証と認可の違い」メモ

## 受け入れ基準
- [ ] ログイン/ログアウトが画面から行える（セッション認証）
- [ ] 未ログインで編集ページに行くとログインへ飛ばされる
- [ ] 他人の記事を編集/削除しようとすると403
- [ ] 認可ロジックがコントローラーの生の `if` に散らばっていない

## 学びの確認（口頭・AI禁止課題）
- トークン認証（API）とセッション認証（画面）は、それぞれどんな時に使うか？
- 認証と認可の違いを説明できるか？
